## Зона ответственности
Папка `server` отвечает за серверную часть проекта на Raspberry Pi 5:
- backend;
- API;
- веб-интерфейс;
- БД;
- аутентификацию и роли;
- блог;
- поддержку;
- админ-панель;
- синхронизацию узлов;
- системные сервисы;
- журналы и defensive-безопасность.

## Базовый стек по умолчанию
Используй реалистичный стек, если пользователь не задал другой:
- Python;
- FastAPI или эквивалентный практичный backend-фреймворк;
- SQLite для MVP как основную БД;
- Nginx для reverse proxy, статики и локального портала;
- WebSocket или SSE для чатов;
- systemd для запуска сервисов;
- Redis только если он реально нужен и это обосновано.

## Главные серверные цели
Нужно реализовать главный сервер проекта, который:
- работает на Raspberry Pi 5 под Pi OS;
- является единственным главным сервером системы;
- поднимает основную точку доступа Wi‑Fi;
- обслуживает локальный веб-портал;
- держит чаты, блог, поддержку, аккаунты и админ-функции;
- управляет правами доступа, банами и чёрными списками;
- хранит медиа и метаданные;
- ведёт логи, резервные копии и журналы безопасности;
- координирует синхронизацию с аппаратными узлами.

## Обязательные режимы
### Закрытый режим
При подключении к локальной сети пользователь должен попадать на страницу:
- вход;
- подача заявки.

Поля заявки:
- имя;
- фамилия;
- номер телефона;
- электронная почта;
- класс / группа;
- флаг учится ли в школе 1575 или его эквивалент.

В закрытом режиме свободной регистрации быть не должно.

### Открытый режим
При подключении к локальной сети пользователь должен видеть:
- вход;
- регистрацию;
- гостевой вход.

При регистрации требуются:
- номер телефона;
- логин;
- пароль.

Ограничения:
- одно устройство не может создать более одного аккаунта;
- один номер телефона не может использоваться более одного раза;
- гостевой вход разрешён только для веб-интерфейса;
- аппаратные клиенты не входят как гости.

### Переключение режимов
При смене режима:
- можно перезапускать только сетевую часть и политики доступа;
- нельзя опасно завершать сервер;
- нужно явно разделять данные закрытого и открытого режимов, если модель данных требует изоляции.

## Обязательные разделы веб-интерфейса
Нужно поддерживать вкладки:
1. Чат
2. Блог
3. Поддержка
4. Аккаунт
5. Устройства
6. Об авторах

### Чат
Поддержать:
- общие чаты;
- приватные или пользовательские чаты;
- создание собственных чатов;
- историю сообщений;
- отправку сообщений;
- отображение участников.

Для приватных комнат поддержать:
- название;
- описание;
- аватар;
- список участников;
- необязательный 4-значный пароль.

Ограничения:
- обычный пользователь — не более 5 пользовательских чатов;
- администратор — без ограничения;
- гость не может создавать чаты;
- администратор технически должен иметь доступ ко всем чатам.

### Блог
Поддержать:
- публикацию постов администраторами;
- список постов;
- просмотр постов пользователями.

### Поддержка
Поддержать:
- личные диалоги пользователя с администратором;
- очередь обращений;
- статусы обращений;
- интерфейс ответа.

### Аккаунт
Поддержать:
- смену аватара;
- базовые настройки профиля;
- отображение ограничений аккаунта.

### Устройства
Поддержать каталог устройств, где для каждого устройства доступны:
- архив или прошивка;
- краткое описание;
- кнопка «У меня есть это устройство»;
- инструкция по установке;
- инструкция по первичной привязке;
- инструкция по смене комбинации входа.

## Аутентификация и доступ
Нужно поддерживать:
- локальные аккаунты;
- логин/пароль;
- роли: гость, пользователь, админ;
- бан;
- чёрный список устройств;
- временную блокировку с причиной и таймером;
- привязку аппаратных комбинаций действий к конкретному устройству;
- повторный полный вход для сброса комбинации.

## RFID / NFC (DEPRECATED in v1.00.00)
Сервер должен поддерживать PN532.
Нужно реализовать:
- централизованное хранение списка карт;
- запись и удаление карт из админского интерфейса;
- использование карт для включения или разблокировки серверного режима.

Не обещай криптографию выше той, которую реально поддерживает выбранный сценарий RFID.

## Безопасность
Серверная часть обязана реализовать defensive-защиту:
- журналы событий безопасности;
- логирование подозрительных запросов;
- rate limiting;
- защиту от brute force;
- ограничение попыток входа;
- блокировку по IP;
- блокировку по device fingerprint в допустимых пределах;
- изоляцию сессий;
- резервные копии БД;
- режим деградации при атаке;
- безопасное выключение;
- оповещение администратора;
- сбор только законной технической информации об инциденте.

Допускаются defensive-ловушки:
- приманочные URL;
- фальшивые endpoint-маршруты;
- отдельные журналы инцидентов;
- корреляционные маркеры.

Запрещено:
- offensive-действия;
- попытки незаконной деанонимизации;
- ложные обещания полной защиты от Wi‑Fi deauth.

## Сетевые и системные правила
- Главный AP поднимает Raspberry Pi.
- Не перекладывай ключевую серверную логику на вспомогательные контроллеры.
- Проектируй безопасное выключение через корректное завершение записи БД и выгрузку буферов.
- При нагрузке проектируй очередь, деградацию и контроль лимитов, а не аварийный отказ.

## Как писать серверный код
- Делай модули небольшими и читаемыми.
- Всегда отделяй конфигурацию, модели, API, сервисы и слой хранения.
- Не смешивай бизнес-логику с веб-слоем без причины.
- Обрабатывай ошибки и пиши логи.
- Не оставляй ключевые места пустыми заглушками.
- Если функциональность велика, режь её на рабочие фазы.

## Самопроверка перед выдачей серверного результата
Проверь:
- запускаемость кода или честно укажи, что реально не запускалось;
- согласованность маршрутов, моделей и зависимостей;
- корректность конфигурации;
- структуру БД и хранение файлов;
- обработку ошибок и логирование;
- риски регрессий.

## Что нельзя делать в этой папке
- превращать аппаратных клиентов в главный сервер;
- строить серверную логику на ложных аппаратных возможностях;
- выдавать схему без реальной структуры файлов;
- заявлять о тестах, которых не было;
- делать большие несвязанные заготовки вместо рабочих модулей.
