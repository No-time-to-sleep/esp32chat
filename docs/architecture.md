# RPi-Only архитектура

Документ фиксирует архитектурную рамку этапа `v1.00.00` для локального распределённого чат-сервера.

> **v1.00.00**: проект перешёл на RPi-Only архитектуру. Внутренние контроллеры исключены из активного состава. Их код сохранён в репозитории как deprecated.

## 1) Архитектурные принципы

- Raspberry Pi 5 — единственный главный сервер и главный AP.
- Внутренние контроллеры разгружают или обслуживают систему, но не подменяют сервер. (deprecated в v1.00.00)
- I2C используется только внутри корпуса и только для лёгких служебных данных. (deprecated в v1.00.00)
- Внешние и удалённые узлы работают через Wi-Fi/BLE/ESP-NOW/USB serial.
- M5Tab получает структурированные данные и отрисовывает UI локально (не видеотерминал). (deprecated в v1.00.00)

## 2) Слои системы

### A. Основной слой (обязательный)

- Главный сервер на Raspberry Pi.
- Веб-портал, API, БД, журналирование, auth, политики доступа.
- Координация узлов, режимов и синхронизации.

### B. Внутренний сервисный слой (внутри корпуса) — deprecated в v1.00.00

- ESP32-S3: watchdog, сервисная телеметрия, аварийные функции.
- 3x M5Stamp S3: малые служебные задачи и сигнализация состояния.
- Atom S3: индикаторный и быстрый сервисный интерфейс.
- PN532: RFID/NFC-доступ по серверной политике.
- M5Tab: локальная HMI админа.
- Встроенный M5Cardputer: локальная сервисная консоль.

### C. Внешний клиентский слой — deprecated в v1.00.00

- M5Cardputer Adv, внешний M5Cardputer, M5StickC Plus 2, T-Embed, Flipper Zero.
- Базовый профиль: текстовый чат + блог + безопасный вход.
- Расширения (медиа/автономность) только при аппаратном подтверждении.

### D. Дополнительные локальные edge-узлы — deprecated в v1.00.00

- Поднимаются только по команде администратора через M5Tab.
- Используют только: `3x M5Stamp S3` и `1x ESP32-S3`.
- Максимум узлов равен числу фактически доступных контроллеров этих типов.
- Edge-узлы не заменяют роль Raspberry Pi и дают ограниченный профиль.

## 3) Главные роли узлов (одна главная роль на узел)

| Узел | Главная роль |
| --- | --- |
| Raspberry Pi 5 | Центральный сервер и координатор системы |
| ESP32-S3 USB-OTG | Внутренний сервисный контроллер и watchdog (deprecated в v1.00.00) |
| M5Stamp S3 (каждый) | Лёгкий внутренний вспомогательный контроллер (deprecated в v1.00.00) |
| Atom S3 | Статус/аварийный индикатор и быстрые действия (deprecated в v1.00.00) |
| M5Tab | Локальная административная HMI-панель (deprecated в v1.00.00) |
| Встроенный M5Cardputer | Локальная сервисная консоль (deprecated в v1.00.00) |
| Внешние handheld-устройства | Пользовательские клиентские интерфейсы |
| PN532 | RFID/NFC-модуль контроля доступа (deprecated в v1.00.00) |

## 4) Каналы связи и данные

### Внутри корпуса

- I2C: heartbeat, статусы, команды, лёгкая телеметрия, сигналы периферии. (deprecated в v1.00.00)
- USB serial/Type-C: служебные прямые каналы (включая M5Stamp S3). (deprecated в v1.00.00)
- UART/USB serial: приоритетный внутренний канал для более тяжёлых пакетов и буферизованной передачи, где I2C не подходит.

### Снаружи корпуса

- Wi-Fi/BLE/ESP-NOW/USB serial: клиентский и служебный обмен.
- Дополнительные edge-узлы публикуются как локальные сетевые сервисы (не loopback localhost).

### Что запрещено через I2C (deprecated в v1.00.00)

- медиа-потоки;
- большие файлы;
- массивные логи;
- каналы с тяжёлой постоянной нагрузкой.

Для тяжёлой внутренней передачи используются UART/USB serial и/или Wi-Fi API-путь, а не I2C. (deprecated в v1.00.00)

## 5) Логическая модель компонентов

### Серверные подсистемы Raspberry Pi

- API + Web UI.
- Auth/roles/mode policies.
- Chat/blog/support/account/devices/admin.
- Device registry + module map + capability gating.
- Security logs + incidents + backup + safe shutdown.

### Подсистемы устройства

- Device profile.
- Transport adapter.
- Local queue + event log.
- Sync client (идемпотентная отправка, retry, conflict policy).

## 6) Обнаружение модулей и пересчёт возможностей

- На старте и в работе выполняется hardware discovery.
- Формируется карта модулей с типом, статусом, ролью и доступностью.
- Пересчитываются лимиты edge-узлов и доступные функции.
- При отсутствии hard-critical модуля система уходит в безопасный hold-state без error-spam.
- При отсутствии soft-critical модуля включается частичная деградация с понятным статусом.

## 7) Устойчивость и деградация

- Потеря внешней сети не выключает основной слой Raspberry Pi.
- При перегрузке применяются лимиты, очереди и режим деградации.
- Незавершённые действия сохраняются в event queue для последующей синхронизации.
- Безопасное выключение всегда приоритетнее быстрого отключения питания.

## 8) Почему архитектура реалистична

- Каждому узлу назначена одна основная роль без дублирования главного сервера.
- M5Tab не создаёт отрицательную нагрузку, потому что не тянет видеовывод Raspberry Pi. (deprecated в v1.00.00)
- Дополнительные узлы ограничены фактическим железом и не выдают недостижимые обещания.
- Ограничения по автономности и медиа зависят от реального хранилища и ресурсов устройств.

## 9) Примечание об архитектурных изменениях

Внутренние контроллеры исключены из архитектуры (v1.00.00). Их код сохранён в репозитории как deprecated.
