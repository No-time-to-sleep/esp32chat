# Session Context

Этот файл — блок 1 из системы контекста `30/30/40`.
Он хранит постоянный контекст проекта: неизменные правила, формат версий, порядок восстановления и правила обновления остальных блоков.

## Система контекстных блоков `30/30/40`

- Блок 1, `30%`: постоянный контекст. Это `SESSION_CONTEXT.md`, корневой [project_agents.md](/D:/project/project_agents.md) и локальные [server/server_agents.md](/D:/project/server/server_agents.md) или [firmware/firmware_agents.md](/D:/project/firmware/firmware_agents.md) по зоне работы.
- Блок 2, `30%`: контекст текущего этапа. Это [STAGE_CONTEXT.md](/D:/project/STAGE_CONTEXT.md). Здесь хранится всё, что одинаково для всех подэтапов текущего этапа.
- Блок 3, `40%`: контекст текущего подэтапа. Это [SUBSTAGE_CONTEXT.md](/D:/project/SUBSTAGE_CONTEXT.md). Здесь хранится только активная версия, её задача, состояние, проверки и следующий шаг.

Проценты — это целевая пропорция внимания и памяти, а не жёсткий подсчёт токенов.

## Порядок восстановления контекста

1. Прочитать `SESSION_CONTEXT.md`.
2. Прочитать [project_agents.md](/D:/project/project_agents.md).
3. Определить рабочую зону: корень, `server` или `firmware`.
4. Если работа идёт в `server` или `firmware`, прочитать локальный `*_agents.md`.
5. Прочитать [STAGE_CONTEXT.md](/D:/project/STAGE_CONTEXT.md).
6. Прочитать [SUBSTAGE_CONTEXT.md](/D:/project/SUBSTAGE_CONTEXT.md).
7. Открывать [PLAN.md](/D:/project/PLAN.md) точечно, только для проверки roadmap, переходов между стадиями и деталей, которых нет в блоках 2 и 3.

## Правила обновления блоков

- Блок 1 меняется редко и только когда меняются глобальные правила, структура проекта или порядок работы между сессиями.
- Блок 2 обновляется при переходе на новый этап или при изменении данных, общих для всех подэтапов этапа.
- Блок 3 обновляется на каждом новом подэтапе и считается главным рабочим снимком текущего шага.
- Перед началом нового шага активный контекст очищается: из блоков 2 и 3 удаляются пункты, которые были нужны только на прошлом шаге и не влияют на текущую задачу.
- В активных блоках держится только то, что нужно для текущего этапа и подэтапа; временные заметки и закрытые вопросы не накапливаются.
- `PLAN.md` остаётся полным roadmap-источником, но не должен подменять активные блоки 2 и 3.
- Если подэтап закрыт, его краткий итог переносится в блок 3 перед переходом к следующему подэтапу.
- Если меняется stage, сначала обновляется блок 2, потом блок 3.

## Формат версий

- Используется формат `v1.00.xx` (RPi-Only архитектура).
- `1` — номер релиза (RPi-Only цикл).
- `00` — номер крупной стадии.
- `xx` — номер подстадии внутри крупной стадии.

Главное правило:

- одна версия = одна законченная подстадия;
- одна версия = одно сообщение;
- крупные устройства и большие блоки обязательно режутся на несколько подстадий;
- нельзя перепрыгивать через стадии без явного объяснения.

## Ключевые постоянные правила

- План всегда ведётся маленькими, проверяемыми версиями.
- Нельзя выдумывать результаты тестов, запусков, аппаратных проверок или публикаций.
- Если среда не позволяет что-то реально проверить, это нужно явно помечать как непроверенное.
- Raspberry Pi 5 остаётся единственным главным сервером системы.
- (DEPRECATED) I²C и внутренние контроллеры исключены из архитектуры. Код сохранён, но не используется.
- Внешние клиентские устройства используют Wi‑Fi, BLE, USB Serial.
- Устройства без накопителя не получают фиктивную глубокую офлайн-автономность.
- Гостевой вход допустим только для веб-интерфейса.
- Defensive-безопасность допустима, offensive-сценарии запрещены.
- Если требование конфликтует с физикой, питанием, радио, теплом, памятью, UX или законностью, нужно предлагать реалистичную замену.

- Raspberry Pi 5 — единственный узел системы. Внутренние контроллеры исключены из архитектуры.
- Стабильность: сервер должен работать 2+ недели без сбоев. При включении питания запускается автоматически.

## Правило "продолжай"

Если после вопроса, просьбы проверки или просьбы подтверждения пользователь отвечает чем-то вроде:

- `продолжай`;
- `continue`;
- `иди дальше`;
- `делай дальше`;
- или любым похожим коротким сообщением,

то работа не блокируется.

В таком случае:

- невыясненные, но не критичные вопросы переносятся дальше;
- непроверенные вручную пункты помечаются как ожидающие проверки;
- опасные, необратимые, аппаратно-зависимые или требующие реального подтверждения шаги не считаются подтверждёнными;
- вместо этого выполняется следующая безопасная подстадия по плану.

## Правило "статус"

Если пользователь пишет что-то вроде:

- `статус`;
- `status`;
- `что осталось`;
- `что проверить`;
- или похожий запрос состояния,

то нужно выдать список:

- всех открытых вопросов;
- всех отложенных подтверждений;
- всех плановых тестов и проверок по порядку;
- всех мест, где работа продолжена без ручного подтверждения пользователя.

## Правило рабочей папки сессии

Если пользователь явно задаёт рабочую папку для текущей сессии, работа ведётся только в этой папке до нового явного указания пользователя.

В таком режиме:

- не сканировать и не изменять файлы вне выбранной папки;
- использовать внешние пути только если пользователь явно попросил это отдельным сообщением.
